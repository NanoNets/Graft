How it works

  1. 1 Point it at your repo. It builds .context/ — one markdown file per system, API, or concept, linked together. The files are the nodes, the links are the edges. That's the whole graph.
  2. 2 Commit it. The graph is just files in git, so git is the sync. Anyone who clones the repo has it, and CI keeps it fresh — when a change moves something, you see it in the graph diff right next to the code, in the same PR.
  3. 3 Your agent reads it before doing work — so it starts knowing the codebase instead of rediscovering it from scratch every time.

Setup

# in your repo
context-graph init          # builds .context/ from your code
git add .context && git commit

# keep it honest — fails CI if the graph is stale
context-graph check

That's it. Anyone who clones the repo gets .context/ for free. Point your agent at it and it reads the graph before working.

The proof

Same model, same tools, same tasks. The only difference is whether the agent got the graph first. On a real 344-file repo it cut tool calls 31%, cost 23%, and latency 17%.

The graph isn't magic — it's just that reading a few linked files beats rediscovering the codebase from scratch every single time.

Runs locally

Everything runs on your machine. No accounts, no cloud calls, no keys. Building the graph uses a local model (Ollama); if you'd rather use a cloud model for better summaries, set a key and it'll use it. Local stays the default.
