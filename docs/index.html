How it works

Push builds it. Clone pulls it. Your agent reads it.

No pipelines, no server to run. The graph is files in the repo, so git does the syncing.

01

Point it at your repo

It builds .context/ — one markdown file per system, API, or concept, linked to each other. The files are the nodes, the links are the edges.

$ context-graph init
344 files → 41 entities · written to .context/

02

Commit it

The graph is just files in git, so git is the sync. Anyone who clones has it, and CI keeps it fresh — you see the graph diff next to the code, in the same PR.

$ git add .context
auth-service.md +3 −1 · alongside the code

03

Your agent reads it

Before doing work, from the CLI or any agent over MCP — so it starts knowing the codebase instead of rediscovering it from scratch.

? how does auth work
reads auth-service.md → api-gateway

Quickstart

Set it up in about a minute.

# in your repo
context-graph init          # builds .context/ from your code
git add .context && git commit

# keep it honest — fails CI if the graph is stale
context-graph check

Everything runs on your machine — no accounts, no cloud calls, no keys. Building the graph uses a local model (Ollama); set OPENROUTER_API_KEY to use a cloud model for better summaries instead. Local stays the default.
